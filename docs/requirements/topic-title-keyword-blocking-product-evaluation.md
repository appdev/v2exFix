# 主题与节点屏蔽产品需求评估

- 日期：2026-05-16
- 执行者：Codex
- 角色：产品经理
- 评估对象：设置中新增“主题屏蔽”，支持按标题关键字和指定节点隐藏主题，并在主题列表顶部提示“已屏蔽 X 个主题”
- 评估依据：Web 首页实测、首页代码、主题列表代码、全部节点接口、现有设置与屏蔽列表能力

## 1. 首页现状观察

首页是 V2Fun 的核心浏览入口，当前结构为：

- 顶部导航：左侧头像/登录入口，中间搜索框，右侧发帖按钮。
- 横向频道 tab：默认包括“最近、最热、技术、创意、好玩、Apple、酷工作、交易、城市、问与答、全部、R2、VXNA、节点、关注、刚更新”。
- tab 管理入口：频道栏右侧有排序/管理入口，进入 `SortTabs`。
- 主题列表：大多数 tab 展示 `TopicItem`，每个主题包含作者、节点、标题、赞同数、回复数、活跃时间、最后回复人等。
- 节点选择能力：`SearchNodeScreen` 已基于 `k.node.all` 提供节点搜索，`SortTabsScreen` 已用它添加首页节点 tab。
- 侧边抽屉：未登录态展示“点我登录”；抽屉中包含外观切换、历史最热、最近浏览、节点导航、更多选项。登录后会增加节点收藏、主题收藏、特别关注、未读提醒等入口。
- 宽屏/平板体验：左侧为首页列表，右侧预留详情区域；未选择详情时显示空状态。

实测状态：

- `http://localhost:19006/Home` 可以展示真实 V2EX 主题列表。
- 首页列表当前可快速扫读大量主题，标题和节点是用户决定是否点开的主要信息。
- 未登录状态下也能完整浏览公开主题列表，因此本地主题屏蔽不应依赖登录。
- `https://www.v2ex.com/api/nodes/all.json` 当前可访问，返回 200；本次验证样本返回 1347 个节点。

## 2. 需求价值评估

结论：建议实施，优先级 P1。

理由：

- 首页内容密度高，用户主要靠标题和节点快速筛选；按标题关键字和指定节点屏蔽能直接降低重复噪音。
- V2EX 热门/最近列表中常出现高回复但用户不感兴趣的话题或节点，例如情感、推广、AI 中转站、设备促销、招聘交易等；这些内容容易跨多个 tab 重复出现。
- 现有 V2EX “忽略主题”是单个主题级操作，必须进入详情或依赖站点能力；本地规则解决的是“一类标题模式”或“某些节点持续不感兴趣”的问题，价值不同。
- 功能可以完全本地化，不影响账号状态，不增加服务端写入依赖，未登录用户也能使用。

## 3. 影响范围评估

强相关列表：

- 首页“最近”：分页主题列表，信息流核心入口。
- 首页普通 tab：最热、技术、创意、好玩、Apple、酷工作、交易、城市、问与答、全部、R2、节点、关注、刚更新。
- 首页自定义节点 tab：使用节点主题分页列表。
- 独立节点主题页、用户主题页、我的主题收藏、我的关注、历史最热、搜索结果等复用 `Topic` 的列表。

低相关或不建议首期覆盖：

- VXNA：模型为 `Xna`，不是 `Topic`，首期不建议覆盖。
- 主题详情页：用户主动进入后不应再屏蔽内容。
- 回复、通知、节点、会员列表：不属于主题浏览流。

对首页的主要影响：

- 列表顶部会新增“已屏蔽 X 个主题”提示，必须避免抢占搜索和频道 tab 的注意力。
- 当前列表可能因过滤变空，需要保留“查看被屏蔽主题”的逃生通道。
- 分页加载后数量会变化，提示需要随当前已加载数据更新。
- 同一主题可能同时命中标题关键字和节点规则，查看时需要展示命中原因，帮助用户判断误伤来源。

## 4. 产品方案建议

### 4.1 功能定位

建议入口仍命名为“主题屏蔽”，副文案为“按标题关键字和节点隐藏主题”。

该功能应明确为本地过滤：

- 不同步到 V2EX。
- 不调用 `ignore/unignore`。
- 不进入现有“忽略主题”服务端列表。
- 删除本地规则后立即恢复列表显示。

### 4.2 设置入口与规则管理

建议在设置页新增入口，位置放在：

1. “主题设置”之后。
2. “屏蔽列表”之前或附近。

原因：

- “主题设置”偏视觉主题，容易和“主题屏蔽”文字混淆；需要相邻但文案清晰。
- “屏蔽列表”是站点级屏蔽，放在附近便于用户理解屏蔽相关能力，但不应合并到同一页首期实现。

规则管理页建议分为两个区块：

- 标题关键字：用户手动输入关键字，添加后以 chip 展示；每个 chip 带删除按钮。
- 屏蔽节点：用户点击“添加节点”进入节点选择器，从全部节点中搜索并多选；确认后以 chip 展示；每个 chip 带删除按钮。

这里需要明确区分新增方式：

- 主题标题关键字是输入型规则，用户自己输入文本。
- 节点屏蔽是选择型规则，用户从 `k.node.all` 的节点列表中选择，不能通过手动输入节点名直接保存。

### 4.3 节点选择设计

项目已有全部节点数据源：

- `servicies/node.ts` 的 `k.node.all` 请求 `/api/nodes/all.json`。
- `screens/SearchNodeScreen.tsx` 已用该数据源实现单选搜索。
- `App.tsx` 已在启动阶段预取 `k.node.all`。

本需求建议新增或扩展节点选择器为多选模式：

- 搜索字段沿用现有逻辑：`title`、`title_alternative`、`name`、`aliases`。
- 节点列表项展示 `title / name`。
- 已选节点有明确选中态。
- 底部或导航栏提供“完成”操作，一次提交多个节点。
- 已选择过的节点再次出现时应保持选中或不可重复添加。

### 4.4 首页提示设计

建议在主题列表内容区顶部展示一条轻量提示：

- 文案：`已屏蔽 X 个主题`
- 样式：小字号、弱强调、可点击；不要使用强警告色。
- 位置：频道 tab 下方、列表第一条主题上方。
- 交互：点击从底部拉起“已屏蔽主题” bottom sheet。
- 统计口径：标题关键字和节点规则合并统计，按当前已加载主题去重。

不建议：

- 不建议使用 toast，因为它只是一瞬间提示，不能提供可追溯入口。
- 不建议放在导航栏或 tab 栏，因为屏蔽数量属于当前列表，而不是全局状态。
- 不建议把屏蔽主题永久计入数据库或历史记录，避免状态复杂化。

### 4.5 查看被屏蔽主题

采用 bottom sheet，标题为“已屏蔽主题”，最高高度为屏幕高度的 90%。

选择 bottom sheet 的原因：

- 保留用户所在 tab 和列表滚动上下文，关闭后能直接回到原列表。
- 更符合移动端临时查看内容的操作习惯。
- 当前查看范围只是“当前列表当前已加载数据”，不需要承载为全局页面。

首期查看范围建议为“当前列表当前已加载数据”，而不是全站屏蔽历史：

- 和“已屏蔽 X 个主题”的统计口径一致。
- 不需要维护额外持久化缓存。
- 用户能快速判断标题关键字或节点是否误伤。

bottom sheet 应复用主题列表项样式，并允许进入主题详情。每条主题需要展示命中原因，例如“命中：关键字 AI”或“命中：节点 酷工作”。点击主题详情时关闭 bottom sheet，再进入主题详情页；用户也可以通过下滑、点击遮罩或关闭按钮返回原列表。

## 5. 验收口径

首期完成标准：

- 用户可在设置页新增、删除、清空标题关键字。
- 用户可从全部节点中搜索并多选屏蔽节点。
- 标题关键字和屏蔽节点都以可删除 chip 展示。
- 关键字和节点规则持久化，重启后仍生效。
- 首页“最近”和普通 tab 中，标题命中或节点命中的主题从可见列表中隐藏。
- 列表顶部显示当前已加载数据的去重屏蔽数量。
- 点击提示可通过 bottom sheet 查看当前被屏蔽主题、命中原因，并能进入详情。
- 未登录用户可使用。
- 站点级“忽略主题”和“屏蔽用户”不受影响。

推荐首期覆盖范围：

- 必须覆盖 `TopicItem` 相关主题列表。
- 首期可暂不覆盖 `XnaItem`。
- 搜索结果如果模型结构接近 `Topic`，建议一起覆盖；如果实现成本偏高，可作为第二阶段。

## 6. 风险与注意事项

- 命名混淆：现有“主题设置”是外观主题，“忽略主题”是站点级屏蔽，新功能叫“主题屏蔽”容易产生歧义。必须用副文案和空状态说明限定为“标题关键字和节点”。
- 新增方式混淆：标题关键字是手动输入，节点是搜索选择；节点不能做成纯文本输入，否则容易出现无效 name 和展示不一致。
- 误伤风险：包含匹配容易误伤，例如“AI”会屏蔽大量正常技术主题；节点屏蔽也可能隐藏大量内容。必须提供查看被屏蔽主题和快速删除 chip。
- 空列表风险：如果用户规则过宽，列表可能为空。空状态需要同时保留屏蔽提示，否则用户会误以为网络或数据异常。
- 数量口径风险：分页列表只知道当前已加载数据，不知道未加载页面。文案不能暗示全站总屏蔽数。
- 性能风险：过滤应在已加载数组上做轻量计算，避免每个列表重复复杂逻辑。
- 移动端空间风险：设置页 chip 需要自动换行，避免长关键字或长节点标题撑破布局。

## 7. 实施建议

建议做成一套通用主题过滤工具，而不是在每个页面复制逻辑：

- 持久化 atom：保存标题关键字数组。
- 持久化 atom：保存屏蔽节点 `name` 数组。
- 纯函数：输入 `Topic[]`、关键字和节点名，输出 `{ visibleTopics, blockedTopics }`，并为 blocked topic 附带命中原因。
- 通用提示组件：接收 `blockedTopics`，负责展示“已屏蔽 X 个主题”和打开 bottom sheet。
- 通用 bottom sheet：接收当前被屏蔽主题列表和命中原因。
- 节点选择器：复用 `k.node.all`，支持搜索和多选。
- Chip 组件：支持文本、可选图标、删除按钮、自动换行。

建议阶段：

1. 第一阶段：设置页规则管理、标题关键字输入、节点多选、chip 删除、首页最近/普通 tab/节点 tab、bottom sheet 查看当前被屏蔽主题。
2. 第二阶段：独立节点页、我的收藏、用户主题、我的关注、历史最热、搜索结果。
3. 第三阶段：根据用户反馈评估是否增加作者维度屏蔽，不建议首期加入。

## 8. 产品结论

该需求适合进入实现阶段，但应控制边界：

- 做“本地标题关键字 + 指定节点屏蔽”，不做全站同步。
- 标题关键字由用户手动输入。
- 节点由用户从全部节点中搜索选择，支持多选。
- 做“当前列表已加载数据”的透明提示，不做全局统计。
- 做“可查看被屏蔽主题”的 bottom sheet 逃生机制，避免误伤不可见。
- 首期不要扩展到作者、正文和回复，避免需求膨胀。

综合评估：

- 用户价值：高。
- 实现复杂度：中。
- 对现有体验风险：中低，可通过轻量提示、chip 删除和查看入口控制。
- 推荐优先级：P1。
