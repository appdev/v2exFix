# 主题与节点屏蔽需求文档

- 日期：2026-05-16
- 执行者：Codex
- 角色：产品经理
- 关联模块：设置页、主题列表、节点搜索、本地偏好存储、被屏蔽主题 bottom sheet

## 1. 背景与目标

V2Fun 当前已有 V2EX 站点级“屏蔽用户”和“忽略主题”能力，入口位于设置页的“屏蔽列表”。这些能力依赖 V2EX 登录态和服务端返回的 `ignored_topics`、`blocked` 数据，解析主题列表时会直接剔除对应主题。

本需求新增本地“主题屏蔽”能力，包含两类规则：

- 标题关键字屏蔽：用户手动输入关键字，客户端隐藏标题命中的主题。
- 指定节点屏蔽：用户从全部节点中搜索并选择一个或多个节点，客户端隐藏这些节点下的主题。

目标：

- 降低用户在浏览主题列表时反复看到不感兴趣标题或节点内容的频率。
- 不依赖 V2EX 服务端，不影响账号级“忽略主题”状态。
- 让用户能追溯当前列表被隐藏了哪些主题，避免误屏蔽后无法发现。
- 在设置页用可删除 chip 管理规则，降低删除和检查成本。

## 2. 术语定义

- 主题：项目中的 `Topic` 数据模型，核心字段包含 `id`、`title`、`member`、`node`、`reply_count` 等；列表中主要通过 `TopicItem` 展示。
- 节点：项目中的 `Node` 数据模型，核心字段包含 `name`、`title`、`aliases`、`id` 等；全部节点可通过 `k.node.all` 获取。
- 主题列表：以 `Topic[]` 或分页 `PageData<Topic>` 呈现的列表页面，包括首页普通 tab、最近主题、节点主题、用户主题、收藏主题、关注人的主题、历史最热主题、搜索结果中的主题列表。
- 站点忽略主题：V2EX 服务端能力，用户在主题详情中点击“忽略”后生效；当前项目通过 `blackListAtom.ignoredTopics` 维护本地镜像。
- 标题关键字屏蔽：本需求新增的客户端本地能力，根据用户手动输入的关键字匹配 `Topic.title`。
- 指定节点屏蔽：本需求新增的客户端本地能力，根据用户选择的节点 `name` 匹配 `Topic.node?.name`。
- 已屏蔽数量：当前列表当前已加载数据中，因标题关键字或指定节点规则被隐藏的去重主题数量。

## 3. 用户故事

- 作为用户，我希望在设置中手动添加、删除和查看标题屏蔽关键字，这样我可以控制哪些标题内容不出现在列表中。
- 作为用户，我希望从全部节点中搜索并多选要屏蔽的节点，这样我可以隐藏某些节点下的所有主题。
- 作为用户，我希望关键字和节点都以带删除按钮的 chip 展示，这样我能快速检查和移除规则。
- 作为用户，我希望主题列表顶部提示“已屏蔽 X 个主题”，这样我知道当前列表有内容被过滤。
- 作为用户，我希望点击屏蔽提示后能查看被屏蔽的主题及命中原因，这样我能判断规则是否误伤，并仍然可以进入主题详情。
- 作为用户，我希望站点级“忽略主题”和本地“主题屏蔽”语义清晰区分，这样我不会误以为本地规则会同步到 V2EX 账号。

## 4. 功能范围

### 4.1 设置入口

- 在设置页新增“主题屏蔽”入口，建议放在“主题设置”和“屏蔽列表”附近。
- 进入后展示规则管理页面，页面标题为“主题屏蔽”。
- 副说明建议为“按标题关键字和节点隐藏主题”。
- 标题关键字和屏蔽节点配置均本地持久化，退出 App 或重启后仍保留。

### 4.2 标题关键字管理

- 标题关键字由用户手动输入。
- 用户可以新增关键字。
- 用户可以删除已有关键字。
- 用户可以清空所有关键字，清空前需要二次确认。
- 空关键字、只包含空白字符的关键字不得保存。
- 重复关键字不得重复保存；重复判断默认忽略前后空格和大小写。
- 已添加关键字以 chip 展示，chip 文案为关键字本身，并带删除按钮。
- 关键字列表为空时显示空状态，例如“暂无屏蔽关键字”。

### 4.3 指定节点管理

- 节点必须通过节点选择器选择，不通过手动输入保存。
- 节点选择器复用全部节点数据源 `k.node.all`，支持按节点 `title`、`title_alternative`、`name`、`aliases` 搜索。
- 节点选择器支持多选，用户可以一次选择多个节点后确认。
- 已选择节点以 chip 展示，chip 文案建议为 `节点标题 / name`；空间不足时至少保留 `节点标题` 或 `name`。
- 每个节点 chip 带删除按钮，删除后该节点屏蔽规则立即失效。
- 重复节点不得重复保存；重复判断使用节点 `name`。
- 节点列表为空时显示空状态，例如“暂无屏蔽节点”。

### 4.4 匹配规则

- 标题关键字规则仅匹配主题标题 `title`。
- 指定节点规则仅匹配主题节点 `topic.node?.name`。
- 标题关键字默认不区分大小写。
- 标题关键字默认包含匹配：标题包含任一关键字即命中。
- 关键字保存时去除前后空格，匹配时不修改标题内容。
- 多个关键字、多个节点之间均为“或”关系。
- 同一主题命中标题关键字或指定节点任一规则，即从可见列表中隐藏。
- 主题缺少 `node.name` 时，不命中节点规则，但仍可命中标题关键字规则。
- 不匹配主题正文、回复、作者名、通知、回复列表或 XNA 内容。

### 4.5 列表屏蔽行为

- 主题列表读取到原始数据后，先计算命中本地规则的主题，再将命中的主题从可见列表中隐藏。
- 当前列表存在被隐藏主题时，在列表顶部显示可点击提示：
  - 文案：`已屏蔽 X 个主题`
  - X 为当前列表当前已加载数据中被隐藏的去重主题数量。
- 当前列表没有被隐藏主题时，不显示提示。
- 提示不应遮挡刷新、分页和原有列表头部信息。
- 分页加载更多后，如果新增数据中有命中主题，数量应随当前已加载数据更新。
- 下拉刷新后，应基于刷新后的当前已加载数据重新计算数量。

### 4.6 查看被屏蔽主题

- 点击“已屏蔽 X 个主题”从底部拉起 bottom sheet 查看被屏蔽主题。
- 查看范围为用户点击入口所在主题列表的当前已加载数据。
- bottom sheet 最高高度为屏幕高度的 90%，内容不足时按内容高度展示。
- 标题建议为“已屏蔽主题”，副标题展示来源列表和数量，例如“最近 · 已屏蔽 3 个主题”。
- 内容区为可滚动列表；当屏蔽主题较多时，仅 sheet 内部滚动，不影响背后的主题列表滚动位置。
- 列表项复用现有主题展示样式，至少展示标题、作者、节点、回复数、最后活跃信息等当前数据已有字段。
- 每条被屏蔽主题需要展示命中原因，例如“命中：关键字 AI”或“命中：节点 酷工作”。
- 用户可以点击被屏蔽主题进入主题详情。
- 点击主题详情时关闭 bottom sheet 并进入主题详情页。
- 用户可以通过下滑、点击遮罩或关闭按钮关闭 bottom sheet，关闭后回到原列表位置。
- bottom sheet 为空时显示“暂无屏蔽主题”，但正常情况下入口仅在数量大于 0 时出现。

## 5. 适用页面

必须适用：

- 首页普通 tab 主题列表。
- 首页“最近”主题列表。
- 首页自定义节点主题列表。
- 独立节点主题页。
- 我的主题收藏列表。
- 我的关注中的主题列表。
- 用户主页的主题列表。
- 历史最热主题列表。
- 搜索结果中的主题列表。

暂不适用：

- XNA 列表，因为它使用 `Xna` 模型而非 `Topic`。
- 主题详情页本身。
- 回复列表、通知列表、节点列表、会员列表。
- V2EX 站点级“忽略主题”列表。

## 6. 数据与状态

- 新增本地持久化状态：标题屏蔽关键字列表。
- 标题关键字推荐数据结构：`string[]`。
- 标题关键字推荐存储 key：`topicTitleBlockKeywords`。
- 新增本地持久化状态：屏蔽节点名称列表。
- 屏蔽节点推荐数据结构：`string[]`，只存节点稳定标识 `node.name`。
- 屏蔽节点推荐存储 key：`blockedNodeNames`。
- 展示节点 chip 时，从 `k.node.all` 返回的 `Node[]` 中按 `name` 查找 `title` 等展示字段；如果节点数据暂不可用，则 fallback 展示 `name`。
- 当前仓库已有 `jotai/blockKeywordsAtom.ts`，但其为普通内存 atom；实现时应改为基于 `atomWithAsyncStorage` 的持久化 atom，或新增语义更明确的持久化 atom 后删除未使用的旧 atom。
- 被屏蔽主题列表不需要长期持久化，应由当前列表原始数据、标题关键字和屏蔽节点实时计算。

## 8. 验收标准

- 在设置页可以进入“主题屏蔽”页面。
- 标题关键字由用户手动输入，添加后以可删除 chip 展示。
- 节点由用户从节点选择器选择，支持一次选择多个，添加后以可删除 chip 展示。
- 添加标题关键字后，所有适用主题列表中标题命中的主题不再直接显示。
- 添加屏蔽节点后，所有适用主题列表中该节点下的主题不再直接显示。
- 被隐藏主题数量大于 0 时，列表顶部显示 `已屏蔽 X 个主题`。
- 点击提示后，可以通过 bottom sheet 查看当前列表被隐藏的主题、命中原因，并可以进入主题详情。
- 删除关键字或节点 chip 后，原本因该规则隐藏的主题在列表中恢复显示。
- 清空全部本地规则后，本地主题屏蔽失效，列表不再显示屏蔽提示。
- App 重启后，关键字和节点配置仍然存在并继续生效。
- 本地规则不会调用 V2EX 的 `ignore/unignore` 接口，也不会修改 `blackListAtom.ignoredTopics`。
- 站点级“忽略主题”仍保持现有行为，不出现在普通列表中。

## 9. 边界场景

- 关键字为空或全空格：不可添加。
- 添加重复关键字：不新增重复项，可提示“关键字已存在”。
- 选择重复节点：不新增重复项，可提示“节点已存在”或保持无感。
- 全部节点接口暂不可用：已保存节点 chip fallback 展示 `name`；新增节点选择器显示加载失败或重试。
- 所有主题都被屏蔽：可见列表显示空状态，同时顶部仍显示 `已屏蔽 X 个主题`，用户可进入查看。
- 分页列表中同一主题重复出现：屏蔽数量按主题 `id` 去重。
- 搜索结果可能只有部分 `Topic` 字段：查看屏蔽主题时按已有字段展示，不要求补全详情。
- 未登录用户也可以使用本地主题屏蔽，因为该功能不依赖 V2EX 账号。

## 10. 非目标

- 不新增按作者、回复内容、正文内容的屏蔽规则。
- 不允许通过手动输入节点名保存节点屏蔽规则；节点规则必须来源于节点选择器。
- 不把本地规则同步到 V2EX 服务端或其他设备。
- 不改变 V2EX 站点级“忽略主题”和“屏蔽用户”的接口行为。
- 不对 XNA、回复、通知和节点列表做过滤。

## 11. 调研依据

- `servicies/types.ts` 定义 `Topic` 和 `Node` 模型，`Topic.title` 是标题关键字匹配字段，`Topic.node?.name` 是节点屏蔽匹配字段。
- `servicies/node.ts` 已有 `k.node.all` 查询，接口为 `/api/nodes/all.json`，返回 `Promise<Node[]>`。
- 2026-05-16 通过网络请求验证 `https://www.v2ex.com/api/nodes/all.json` 返回 200，样本返回 1347 个节点。
- `screens/SearchNodeScreen.tsx` 已复用 `k.node.all` 实现节点搜索，搜索字段包括 `title`、`title_alternative`、`name`、`aliases`。
- `screens/SortTabsScreen.tsx` 已通过 `SearchNode` 选择节点并添加到首页 tab，可作为节点选择交互参考，但本需求需要支持多选。
- `App.tsx` 在初始化阶段调用 `k.node.all.useQuery()`，说明全部节点数据已有预取路径。
- `components/topic/TopicItem.tsx` 是多数主题列表复用的主题列表项组件。
- `servicies/helper.ts` 的 `parseTopicItems` 负责从 V2EX HTML 列表解析 `Topic[]`，并会基于服务端 `ignored_topics` 和 `blocked` 直接过滤主题。
- `screens/SettingScreen.tsx` 已有设置页入口模式，当前包含“主题设置”和“屏蔽列表”。
- `screens/BlackListScreen.tsx` 的“屏蔽列表”包含“屏蔽用户”和“忽略主题”，属于站点级屏蔽语义。
- `screens/HomeScreen.tsx`、`screens/NodeTopicsScreen.tsx`、`screens/MyTopicsScreen.tsx`、`screens/MyFollowingScreen.tsx`、`screens/MemberDetailScreen.tsx`、`screens/HotestTopicsScreen.tsx`、`screens/SearchScreen.tsx` 均存在主题列表渲染入口，是后续实现需要覆盖的主要页面。
